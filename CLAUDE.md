# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## What this repo is

A small, repeatable demo/verification harness for validating TypeScript provider bindings generated by CDK Terrain against the AWS Terraform provider. It exercises plugin-protocol features that go beyond baseline resources/data-sources — provider-defined functions, ephemeral resources, list resources, provider actions, and resource identities — and documents what does/doesn't get generated for them.

It is deliberately source-agnostic so the same scenarios can be re-run for ongoing capability testing against any of three cdktn sources:

1. **`latest`** — the current published release line (`cdktn@latest`, currently 0.23.x).
2. **`next`** — the published prerelease line via the mutable `@next` dist-tag (`cdktn@next`, currently 0.24.0-pre.x). **This is the committed default.**
3. **PR head** — a locally built `cdk-terrain` checkout, selected via `CDKTN_REPO` (`file:` dependency).

Switch with `pnpm run use:latest` / `pnpm run use:next` / `pnpm run use:prhead`; `pnpm run assert:deps` verifies the resolved source matches the intended mode.

This is not a library; it has no unit tests. "Correctness" here means: schema captured → bindings generated → examples typecheck → examples synth → `terraform`/`tofu plan` succeeds. Everything is validated by actually running the CDKTF pipeline against a real provider schema and a real plan.

## Required environment before anything works

- `pnpm`, Node.js `>=22.19.0`, plus `mise` with `terraform@1.15.7` and `opentofu@1.12.3` available.
- A cdktn source selected: `package.json` defaults `cdktn`/`cdktn-cli` to the mutable `@next` dist-tag. Run `pnpm run use:latest`/`use:next`/`use:prhead` to switch (the last needs `CDKTN_REPO` pointing at a built `cdk-terrain` checkout with `packages/cdktn` and `packages/cdktn-cli/bundle/bin/cdktn.js`), then `pnpm run assert:deps` to confirm the resolved source matches. `pnpm install` only reproduces the lockfile — it does not advance a locked dist-tag; run `pnpm run refresh:next` (or `refresh:latest`) to deliberately follow the mutable tag to its newest published build. Do not use `pnpm update --latest`: npm's `latest` is still 0.23.x, so `--latest` would jump off the tag rather than follow it.
- `AWS_PROFILE` for Terraform planning (default `tcons-hermes`), or run under `aws-vault exec <profile> --no-session -- ...` — see the `aws-vault` note below if doing the latter.
- `TERRAFORM_BINARY_NAME` — set this to the `mise`-resolved terraform/opentofu binary before running schema/plan commands, e.g. `export TERRAFORM_BINARY_NAME=$(mise x terraform@1.15.7 -- which terraform)`.

`aws-vault` note: the examples read `AWS_ACCESS_KEY_ID` at synth time to decide whether to pass `profile` to `AwsProvider`. Passing `profile` alongside credential env vars injected by `aws-vault exec` makes the AWS Go SDK resolve creds from that profile's `~/.aws/config` block (which aws-vault-managed profiles usually don't have) instead of the injected env vars — so run `synth:all` and `plan:all` inside the *same* `aws-vault exec <profile> --no-session -- ...` invocation, not `AWS_PROFILE=<profile>` plus a separately-invoked plan.

## Commands

Full verification flow (Terraform):

```bash
export AWS_PROFILE=tcons-hermes
export TERRAFORM_BINARY_NAME=$(mise x terraform@1.15.7 -- which terraform)
pnpm install                      # or: pnpm run use:latest | use:next | use:prhead
pnpm run assert:deps              # verify resolved cdktn source matches intended mode
pnpm run schema:aws               # `terraform providers schema -json` -> schema/aws-provider-schema.json
pnpm run generate:aws             # cdktn get -> .gen/
pnpm run inspect:schema            # print schema capability counts + generated-file presence
pnpm run assert:generated-features
pnpm exec tsc --noEmit
pnpm run synth:all                # synth all examples/
pnpm run plan:all                 # terraform init+plan (refresh=false) for all examples/
```

Swap `TERRAFORM_BINARY_NAME` to an OpenTofu binary (`mise x opentofu@1.12.3 -- which tofu`) to re-run `schema:aws`/`synth:all`/`plan:all` against OpenTofu instead — bindings are shared, only the schema capture and plan binary change.

`pnpm run check` is the fast loop while iterating on generated bindings: regenerates from the existing captured schema and synths both examples (`generate:aws && synth:all`) without re-capturing schema or re-planning.

Single-example iteration:

```bash
pnpm run synth:functions-only        # or synth:functions-ephemeral
pnpm run plan:functions-only         # or plan:functions-ephemeral (requires synth first)
```

There is no lint/test script; `tsc --noEmit` is the correctness gate for generated + example code.

## Architecture

- `scripts/cdktn.js` — mode-aware wrapper that every `generate:aws`/`synth:*` script routes through. With `CDKTN_REPO` set it invokes that checkout's built CLI bundle (`packages/cdktn-cli/bundle/bin/cdktn.js`); otherwise it invokes the `cdktn-cli` installed from npm. One invocation path for all three sources.
- `scripts/use-cdktn.js <latest|next|prhead>` — repoints the `cdktn`/`cdktn-cli` dependency and reinstalls. `latest`/`next` write the mutable dist-tag (not a pinned version) into `package.json`; `prhead` writes a `file:` dependency on `$CDKTN_REPO/packages/cdktn`. Exposed as `use:latest`/`use:next`/`use:prhead`.
- `scripts/assert-deps.js` — a guard, not a test: with `CDKTN_REPO` set it requires `cdktn` to resolve from that local checkout (and the CLI bundle to exist); otherwise it requires `cdktn` to resolve from npm (not a `file:`/worktree path) with `cdktn-cli` at a matching version. Prints the resolved version so `latest` (0.23.x) vs `next` (0.24.0-pre.x) is visible without pinning either here.
- `scripts/capture-aws-schema.sh` — spins up a throwaway `terraform init` in a tempdir with only the `hashicorp/aws` provider required, dumps `terraform providers schema -json` to `schema/aws-provider-schema.json`, then runs `inspect-aws-schema.js` on it.
- `scripts/inspect-aws-schema.js` / `scripts/assert-generated-features.js` — read the captured schema and report/assert counts for `functions`, `ephemeral_resource_schemas`, `list_resource_schemas`, `action_schemas`, `resource_identity_schemas` (plus resource/data-source counts), and check `.gen/providers/aws/` for the corresponding generated files/directories. `assert-generated-features.js` is the hard gate (throws on missing bindings for functions/ephemeral/resource-identities); `inspect-aws-schema.js` is informational and also flags via `NOTE:` lines when a schema capability exists but no first-class binding directory was generated for it (list resources / actions / identities today).
- `.gen/providers/aws/` — generated TypeScript bindings, **committed intentionally** (not gitignored) so that diffs between PR heads are directly reviewable in git history. `.gitattributes` marks `.gen/**` and `schema/*.json` as `linguist-generated`.
- `examples/<name>/cdktf.json` + `examples/<name>/main.ts` — each example is its own independent CDKTF app (own `cdktf.json`, own `app` command running `ts-node main.ts` directly against `main.ts`, not through a build step). Adding a new example means adding both files and wiring `synth:<name>`/`plan:<name>` scripts plus updating `synth:all`/`plan:all`.
  - `functions-only`: exercises provider-defined functions only, no provider-level infra assumptions beyond a configured `AwsProvider`. The minimal repro if function-token synthesis regresses. Includes `AwsProviderFunctions.userAgent(...)` in an output.
  - `functions-ephemeral`: adds an ephemeral resource (`EphemeralAwsSecretsmanagerRandomPassword`) referenced via `Fn.ephemeralasnull(...)` in a `sensitive: true` output — this is required because Terraform/OpenTofu reject using an ephemeral value in a non-ephemeral context, and reject a non-sensitive output derived from ephemeral data. Also sets a `lifecycle.postcondition` on the ephemeral resource (referencing the non-sensitive `self.password_length`, not the sensitive `random_password`) to exercise the narrowed `TerraformEphemeralResourceLifecycle` type.
  - `mixins-s3`, `mixins-data-recovery`, `mixins-aspect-instance`: exercise the `IMixin`/`construct.with(...)` primitive (`constructs@10.6.0`, shipped starting `cdktn@0.23.4`) against the proposed docs concept page — provider-agnostic mixin, provider-specific mixin via `addOverride`, multi-resource-type mixin via typed `put*` setters, and the mixin+aspect pairing (`RequireImdsv2` mixin + `ValidateImdsv2` aspect on `Instance`).
  - `l2-kinesis-stream`, `l2-secretsmanager-secret`, `l2-ec2-cidr-split`: port real `aws-cdk-lib` L2 construct logic (not the constructs themselves — hand-translated) onto generated `awscc` (Cloud-Control-derived) bindings + the [`cfncompat`](https://registry.terraform.io/providers/cdktn-io/cfncompat) provider's CloudFormation-intrinsic-function polyfills (`provider::cfncompat::*`). `terraformProviders` in `cdktf.json` includes `"awscc"` and `"cdktn-io/cfncompat"` (both public-registry, no local mirror needed — see README for the disposable `filesystem_mirror` pattern if ever testing an *unpublished* cfncompat build). See README's Example projects section for what each ports and cites (exact upstream file:line), and UX notes for two binding-generation bugs found while building them.
  - Root `cdktf.json` (`app: "echo root generation only"`) exists only so `cdktn get` has a project root to generate `.gen/` into; it is not synthesized itself.

## Known constraints discovered while building this demo (keep respecting these)

- Import generated modules directly (e.g. `./.gen/providers/aws/provider`, `./.gen/providers/aws/provider-functions`), never the AWS provider barrel (`./.gen/providers/aws`) — the barrel pulls in thousands of generated classes and can OOM Node/TypeScript during synth.
- Never call `AwsProviderFunctions.userAgent(...)` (or any provider function) inside the `AwsProvider(...)` configuration block itself. Provider functions are namespaced per-provider (`provider::aws::user_agent`), so calling one while configuring that same provider creates a self-referential evaluation cycle that Terraform/OpenTofu reject. Only use provider functions after provider configuration exists (e.g. in outputs), and keep provider configuration static.
- As of the recorded `next` (0.24.0-pre.x) build, no first-class generated bindings exist for list resources, provider actions, or resource identity schemas, even though the AWS provider schema exposes all three (see README's "Current verification results" for exact counts at the tested version). Don't assume classes for these exist under `.gen/` — re-check on each source.
- Generated ephemeral resources use `TerraformEphemeralResourceLifecycle` (precondition/postcondition only), not the full managed-resource `TerraformResourceLifecycle`. Referencing a sensitive computed attribute (e.g. `random_password`) directly in a lifecycle condition fails Terraform's sensitive-value-in-condition check — reference a non-sensitive attribute instead (see `functions-ephemeral`'s `self.password_length` postcondition).
- Don't pass an explicit `profile` to `AwsProvider` when direct credential env vars (`AWS_ACCESS_KEY_ID`) are already present (e.g. under `aws-vault exec`) — see the `aws-vault` note above.
- `Annotations.addError(...)` inside an `IAspect.visit(...)` makes `app.synth()` throw (not just annotate metadata) — verified via `mixins-aspect-instance`. Useful for reasoning about aspect-based validation elsewhere in this repo.
- The committed default resolves `cdktn`/`cdktn-cli` from the mutable `@next` dist-tag, not a pinned version or a `file:` path. A `file:`/PR-head checkout is a supported *mode* (`pnpm run use:prhead` with `CDKTN_REPO`), not the committed state — don't commit a `file:` dependency or a hard version pin. `pnpm run assert:deps` enforces that whatever mode is selected resolves consistently.
- For current gaps — write-only registration limitations, missing list/action/identity bindings, and no first-class `provider_meta` construct — see README's "Current blockers / gaps" section. The historical PR #296 issues recorded there are implementation context; do not revert the released fixes for ephemeral lifecycle restriction, dynamic provider-function results, or literal `null` provider-function arguments.
